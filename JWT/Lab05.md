#  JWT authentication bypass via kid header path traversal

## 📌 Lab Details
- **Title**: JWT authentication bypass via kid header path traversal
- **Difficulty**: Practitioner
- **Category**: Json Web Token
- **Lab URL**: https://portswigger.net/academy/labs/launch/b798d10d5ebbe86319083eaa1218f27ff9fe579f9e6e02a894077bb0cfe771e0?referrer=%2fweb-security%2fjwt%2flab-jwt-authentication-bypass-via-kid-header-path-traversal

## 🔍 Summary
This lab demonstrates how improper handling of the JWT `kid` (Key ID) header can lead to authentication bypass. The application uses the `kid` header to dynamically load a signing key from the server's filesystem. Because the server fails to sanitize the `kid` value, an attacker can perform path traversal and point the server to predictable files such as `/dev/null`.<br>
By signing the JWT using a null byte secret and setting the `kid` parameter to `/dev/null`, we can forge a valid token for the **administrator** account and gain access to the admin panel.

## 🛠 Steps to Solve
1. Log in as a normal user. After loggin in, capture the request to: `GET /my-account`.
2. Check admin access: `GET /admin`.
3. Generate a new symmetric key using **JWT** editor extension. Replace the generated `k` value with: `AA==`.
4. Modify the `kid` parameter in the JWT header to: `"../../../../../../../dev/null"`. <br> Modify the payload: `"sub": "administrator"`.
5. Sign the token using the symmetric key generated by the **JWT** editor extension.
6. Access the admin panel: `GET /admin`.
7. Locate the delete endpoint: `/admin/delete?username=carlos`.

## 📖 Key Takeaways
- Never trust JWT header parameters such as `kid`.
- Using filesystem paths directly from user-controlled input is dangerous.
- JWT verification logic should only load keys from trusted, hardcoded locations.
